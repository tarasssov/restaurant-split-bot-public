# 2-недельный план: стабилизация качества и подготовка к прод-боту

## Этап 1. Product hardening (Дни 1-4)
Цель: предсказуемый UX и понятные статусы качества.

1. Ввести единый статус `quality_status`:
- `good`
- `warning`
- `low_confidence`

2. Зафиксировать правила:
- `good`: обычный поток.
- `warning`: показываем причины + рекомендуем проверку перед подтверждением.
- `low_confidence`: блок перехода к сплиту, пока пользователь не поправит позиции или не пересканирует.

3. Короткие UX-тексты:
- причина: `synthetic`, `semantic_fail`, `мало позиций`, `аномально крупная позиция` и т.д.
- один формат сообщений во всех ветках (`rule`/`llm`).

4. Критерий готовности:
- в каждом сценарии пользователь видит статус и понимает, что делать дальше;
- нет тихих переходов при плохом качестве.

## Этап 2. Data/observability loop (Дни 5-8)
Цель: собирать данные для тюнинга, а не править вслепую.

1. Лог чек-сессии (JSONL):
- `timestamp`
- `chat_id/session_id` (можно хеш)
- `ocr_provider`, `mode_used`
- `quality_status`, ключевые метрики (`synthetic`, `diff`, `semantic_fail`, и пр.)
- `ocr_text` (или ссылка на файл)
- итоговый список позиций перед подтверждением

2. Лог правок пользователя (JSONL):
- тип правки: `delete`, `rename`, `reprice`, `rescan`
- индекс/старое значение/новое значение
- финальный accepted список

3. Базовый отчёт-агрегатор (скрипт):
- сколько чеков по статусам;
- топ причин `warning/low_confidence`;
- топ частых ручных правок.

4. Критерий готовности:
- по любому проблемному чеку можно восстановить путь `OCR -> parse -> правки -> финал`;
- есть недельная сводка без ручного разбора логов.

## Этап 3. Targeted parser/LLM tuning (Дни 9-14)
Цель: улучшать качество только по подтверждённым частым ошибкам.

1. Взять топ-10 паттернов из логов:
- склейки названий;
- неверные цены `qty+price`;
- пропуски позиций;
- мусорные строки в `item name`;
- ложные `low-confidence`.

2. Для каждого паттерна:
- 1 маленькая правка parser или 1 правка LLM prompt/post-check;
- тест-кейс (fixture + ожидаемый результат);
- регресс на baseline + новых кейсах.

3. Acceptance:
- baseline не ухудшается;
- снижается доля `warning/low_confidence` на реальных чеках;
- уменьшается число ручных правок на чек.

## Что не делаем сейчас
- обучение своей модели;
- смена OCR-провайдера;
- большой рефактор parser;
- сложная ML-инфраструктура.

## Контрольные точки
1. Конец 1 недели: статусы/гейты готовы + логи собираются.
2. Конец 2 недели: топ-10 ошибок частично закрыт, baseline не деградировал, видно улучшение по прод-метрикам.
